1. Introducción

1.1 Propósito

El propósito de este documento es describir la estrategia de pruebas, los tipos de pruebas, los criterios de entrada y salida, el plan de ejecución, los ambientes utilizados y los mecanismos de gestión de defectos para garantizar que la aplicación móvil EDI-301 cumpla con los requisitos funcionales y no funcionales definidos. Este documento sirve como referencia para desarrolladores, personal de QA y responsables del aseguramiento de calidad.

1.2 Visión general del proyecto

El sistema EDI-301 es una aplicación móvil diseñada para gestionar el programa de Equipos de Desarrollo Integral (EDI) de la Universidad Linda Vista. Las funcionalidades principales incluyen:

  • Autenticación y manejo seguro de sesión.
  • Visualización de noticias y publicaciones.
  • Gestión y consulta de familias EDI.
  • Acceso a información de eventos y actividades.
  • Actualización de datos de perfil y carga de imágenes.

Las pruebas descritas en este documento se enfocan en verificar que estas funciones operen correctamente en dispositivos móviles compatibles y que la aplicación se integre adecuadamente con la API backend.

2. Alcance

2.1 En alcance

Las pruebas abarcan los siguientes aspectos del sistema:

Funcionalidades principales

  • Inicio de sesión y uso de tokens de autenticación.
  • Visualización de noticias y publicaciones.
  • Consulta y edición de la información de familia EDI.
  • Carga y visualización de imágenes de perfil y portada.
  • Gestión y consulta de eventos.
  • Navegación entre pantallas y mantenimiento de sesión.
  • Validaciones en formularios dentro de la app.

Pruebas de la API

  • Validación de respuestas y códigos de estado HTTP.
  • Verificación de seguridad y manejo de tokens.
  • Validación de datos de entrada y salida de la API.

Pruebas no funcionales

  • Tiempo de respuesta en pantallas clave.
  • Compatibilidad con versiones seleccionadas de Android y iOS.
  • Usabilidad básica (flujo intuitivo y mensajes claros).

2.2 Fuera de alcance

Quedan fuera del alcance de este plan de pruebas:

  • Pruebas en entornos de escritorio o navegadores web.
  • Pruebas de infraestructura avanzada externa al proyecto (balanceadores, CDN, etc.).
  • Pruebas de carga extrema por encima de 5,000 usuarios concurrentes.
  • Pruebas sobre funcionalidades no implementadas o solo prototipadas.

3. Estrategia de pruebas

La estrategia de pruebas combina pruebas manuales y automatizadas, con foco en la calidad funcional, estabilidad, seguridad y experiencia de uso en dispositivos móviles.

3.1 Objetivos de prueba

  • Verificar que las funcionalidades cumplan los requisitos establecidos.
  • Detectar y corregir defectos antes del despliegue a producción.
  • Asegurar estabilidad, seguridad y usabilidad en dispositivos móviles.
  • Confirmar la correcta integración con la API backend.
  • Garantizar que las actualizaciones no introduzcan errores regresivos.

3.2 Supuestos de prueba

  • Los usuarios contarán con conexión estable a Internet.
  • El servidor backend estará disponible durante las sesiones de prueba.
  • Las credenciales de prueba entregadas serán válidas y operativas.
  • Los dispositivos móviles utilizados cumplen los requisitos mínimos del proyecto.

3.3 Enfoque de datos

Se emplearán los siguientes tipos de datos de prueba:

  • Datos reales controlados: usuarios, familias y eventos creados por QA.
  • Datos simulados: casos límite para validar formatos y reglas.
  • Datos anómalos: entradas incorrectas para comprobar manejo de errores.

3.4 Niveles de prueba

  1. Pruebas unitarias: validan funciones individuales.
  2. Pruebas funcionales: verifican flujos completos en la app.
  3. Pruebas de aceptación de usuario (UAT): validación por el administrador EDI.
  4. Pruebas de regresión: aseguran estabilidad tras cambios.

3.5 Pruebas unitarias

Se aplican sobre funciones aisladas del backend (controladores, servicios, validadores) y sobre lógica interna en la app (providers, services, DTOs).

Herramientas sugeridas: Jest para backend y flutter test para la app.

3.6 Pruebas funcionales

Se realizan de forma manual sobre la aplicación móvil, cubriendo principalmente:

  • Inicio de sesión y cierre de sesión.
  • Navegación entre pantallas principales.
  • Consulta y edición de familia EDI.
  • Carga y visualización de imágenes.
  • Creación y consulta de eventos.
  • Publicaciones y visualización de noticias.

3.7 Pruebas de aceptación de usuario (UAT)

El administrador del programa EDI y otros usuarios clave evaluarán que:

  • Las funciones satisfacen las necesidades del programa.
  • La navegación sea clara y comprensible.
  • La información mostrada sea consistente y confiable.
  • Las operaciones críticas (eventos, publicaciones) funcionen correctamente.

3.8 Pruebas de regresión

Se ejecutarán después de cada actualización importante o corrección crítica, cubriendo funcionalidades como:

  • Autenticación y manejo de sesión.
  • Publicaciones y noticias.
  • Carga y visualización de imágenes.
  • Gestión de eventos.
  • Perfil y datos del usuario.

4. Estrategia de ejecución

4.1 Criterios de entrada

  • Backend desplegado en ambiente de pruebas.
  • Aplicación EDI-301 instalada en los dispositivos de prueba.
  • Datos de prueba configurados en la base de datos.
  • Acceso a herramientas de monitoreo y registro (logs, Postman, etc.).

4.2 Criterios de salida

  • Todos los casos de prueba críticos ejecutados.
  • Defectos de severidad alta corregidos o mitigados.
  • Informe de resultados de pruebas elaborado y revisado.
  • Validación final por parte del administrador del programa EDI.

4.3 Validación y gestión de defectos

Los defectos se registrarán en un sistema de seguimiento (por ejemplo, GitHub Issues) y se clasificarán según su severidad:

  • Crítico: impide el uso de una funcionalidad esencial.
  • Alto: afecta funciones importantes pero con alternativas.
  • Medio: errores que no detienen el flujo pero afectan la experiencia.
  • Bajo: detalles menores o cosméticos.

Flujo básico:

  1. Registro del defecto por QA o usuario clave.
  2. Asignación a desarrollador responsable.
  3. Corrección y despliegue en ambiente de pruebas.
  4. Re-ejecución de la prueba afectada.
  5. Cierre o reapertura del defecto según resultados.

5. Requisitos de entorno

5.1 Ambientes de prueba

Ambiente de QA

  • Dispositivos Android (ej. 8, 10, 12).
  • Dispositivos iOS (ej. 13, 14, 16).
  • API en entorno de pruebas (por ejemplo: https://qa.edi.ulv/api).
  • Node.js versión 18 o superior.
  • SQL Server para la base de datos del sistema.
  • Flutter 3.x para compilación de la app.

Ambiente de producción

  • Dispositivos reales utilizados por alumnos y colaboradores.
  • API oficial de producción del sistema EDI-301.

6. División / área significativamente impactada

Las áreas más impactadas por el proceso de pruebas y la operación del sistema EDI-301 son:

  • Capellanía Universitaria (administración del programa EDI).
  • Departamento de Tecnologías de Información.
  • Estudiantes y empleados que participan en las familias EDI.

7. Dependencias

La correcta ejecución de las pruebas depende de:

  • Disponibilidad de la API y la base de datos del sistema.
  • Acceso a datos de prueba suficientes y representativos.
  • Colaboración del equipo de desarrollo para la corrección de defectos.
  • Documentación técnica actualizada del sistema.
  • Disponibilidad de dispositivos móviles compatibles para pruebas.
